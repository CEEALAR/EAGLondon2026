    Create ceealar-pulse/app/api/import/route.ts as a Next.js route handler exporting POST.

    Implementation requirements:

    1. Auth check: create server client via createClient() from @/lib/supabase/server, call getUser(). If no user, return 401 JSON.

    2. Parse multipart FormData: const formData = await req.formData(); const file = formData.get('file') as File. If no file, return 400.

    3. Read file buffer: const buffer = Buffer.from(await file.arrayBuffer())

    4. Parse XLSX with SheetJS: import * as XLSX from 'xlsx'. Use XLSX.read(buffer, { type: 'buffer' }). Target the 'Attendee Data' sheet by name: const sheet = workbook.Sheets['Attendee Data']. If the sheet is missing, fall back to workbook.Sheets[workbook.SheetNames[0]].

    5. Convert to 2D array: XLSX.utils.sheet_to_json(sheet, { header: 1, range: 4 }) — range: 4 means start at row index 4 (row 5 in 1-indexed, which is the header row). Result[0] is the header array, result[1+] are data rows.

    6. Build header index map: const headerMap: Record<string, number> = {}; row 0 of the result maps header string to column index. This allows dynamic column lookup — do not hardcode column indices.

    7. Helper function sanitize(val: unknown): string | null — return null if val is undefined/null/''. Otherwise String(val).trim().replace(/​/g, '') — return null if empty after sanitize.

    8. Helper splitArray(val: unknown): string[] | null — sanitize first, then split on ';', map each part through sanitize, filter nulls. Return null if result is empty.

    9. Helper seekingWork(val: unknown): boolean | null — sanitize first, then: 'yes'→true, 'no'→false, else null (case-insensitive).

    10. Build upsert rows: iterate result.slice(1) (skip header row). For each row:
        - Extract swapcard_url via headerMap['Swapcard'] index — sanitize it
        - If null/blank, increment skipped counter, skip row
        - Build an attendee object mapping all 15 columns using headerMap lookup + sanitize/splitArray/seekingWork helpers
        - Map: 'First Name'→first_name, 'Last Name'→last_name, 'Company'→company, 'Job Title'→job_title, 'Career Stage'→career_stage, 'Biography'→biography, 'Areas of Expertise'→expertise (splitArray), 'Areas of Interest'→interests (splitArray), 'How Others Can Help Me'→how_others_can_help, 'How I Can Help Others'→how_i_can_help, 'Country'→country, 'Seeking work?'→seeking_work (seekingWork helper), 'Recruitment'→recruitment, 'LinkedIn'→linkedin

    11. Upsert in batches of 100 using service role client (import createClient from @supabase/supabase-js, use NEXT_PUBLIC_SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY). Call .from('attendees').upsert(batch, { onConflict: 'swapcard_url', ignoreDuplicates: false }). Count inserted vs updated: Supabase upsert does not return insert/update distinction separately, so track by comparing data.length — if returning upsert data is not feasible, use a pre-check SELECT count for each batch (too expensive). Simplest: use ignoreDuplicates: false and report total rows processed minus skipped as "inserted or updated". Alternatively, fetch existing swapcard_urls before upsert, then split batches into inserts vs updates.

    Practical simplification (use this): Before upsert, SELECT swapcard_url FROM attendees. Build a Set of existing URLs. Rows in the set = updated; rows not in set = inserted. Perform a single upsert call for all rows. Return { inserted, updated, skipped, errors }.

    12. Return JSON: NextResponse.json({ inserted, updated, skipped, errors: [] })

    The service role client must be created inline (not from lib/supabase/server.ts which uses anon key): import { createClient as createAdminClient } from '@supabase/supabase-js' and instantiate with SUPABASE_SERVICE_ROLE_KEY. This is the same pattern used in app/auth/callback/route.ts per the 01-01-SUMMARY.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /api/import | File upload crosses unauthenticated boundary — route must verify session |
| Import route → Supabase | Service role key grants full table access — must never be exposed to client |
| XLSX file content → DB | Untrusted third-party data (Swapcard export) enters as string values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | data/swapcard.xlsx | mitigate | File is gitignored; migration comment and page UI warn never to commit; .gitignore entry verified before commit |
| T-02-02 | Elevation of Privilege | POST /api/import | mitigate | Route handler calls createClient().auth.getUser() before processing; returns 401 if no session; middleware also blocks unauthenticated access at edge |
| T-02-03 | Tampering | XLSX cell values → DB | mitigate | sanitize() strips zero-width spaces and trims; React renders all string values as text nodes (no dangerouslySetInnerHTML anywhere) — XSS not possible |
| T-02-04 | Tampering | Mass upsert data integrity | accept | Last-write-wins on re-import is acceptable for a 4-person team tool; no concurrent import scenario in practice |
</threat_model>
