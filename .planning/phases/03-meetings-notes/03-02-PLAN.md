<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /api/meetings | JSON body with owner_id crosses trust boundary — server validates session |
| owner_id in body | Client can supply any UUID as owner_id — acceptable for 4-person trusted team tool |
| activity.actor_id | Set from server-side user.id (from auth.getUser()), not from client body — prevents spoofing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Elevation of Privilege | POST /api/meetings | mitigate | Route calls supabase.auth.getUser() before any insert; returns 401 if no session |
| T-03-06 | Spoofing | activity.actor_id | mitigate | actor_id is set from server-side user.id (auth.getUser()), never from the request body; client cannot spoof who acted |
| T-03-07 | Tampering | owner_id in create body | accept | Any authenticated team member can set any other team member as owner — this is desired behavior (e.g. Attila creates meeting and assigns to Greg); trusted 4-person team |
| T-03-08 | Information Disclosure | /meetings list page | mitigate | Page is inside (app) layout which redirects unauthenticated users; supabase.auth.getUser() called at top of page before any data fetch |
</threat_model>
