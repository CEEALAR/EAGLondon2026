## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server → attendee detail page | meetings array fetched server-side; conflict data derived on server; no client-side trust boundary for conflict logic |
| Client → MeetingCreateDialog | wantToMeetOwners passed as server-computed prop; dialog cannot forge or suppress the warning |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Information Disclosure | Conflict banner reveals other team members' meeting intentions | accept | By design — all four team members are co-workers managing the same event; cross-visibility is the goal, not a risk |
| T-05-06 | Tampering | Client-side suppression of conflict banner | accept | Banner is purely informational; suppressing it on the client harms only the user doing so; no security invariant broken |
| T-05-07 | Spoofing | wantToMeetOwners display names | accept | Names come from team_members.display_name via a server-side Supabase join on authenticated session; data is internal and low-risk |
| T-05-08 | Elevation of Privilege | Conflict derivation accessing other members' meeting data | accept | All team members have full read access to all meetings by design (4-person internal tool); this is an intentional data sharing model, not a privilege escalation |
</threat_model>
